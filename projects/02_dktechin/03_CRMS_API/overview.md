# CRMS-API 서버 분리 구축(RESTful) 및 운영

## 개요

- **기간:** 2023.11 ~ 2024.07
- **유형:** 회사 프로젝트
- **인원:** 4명
- **역할:** 기여도 40%

## 프로젝트 소개

> CRMS 내 혼재된 API 기능을 분리하기 위해 RESTful 기반 CRMS-API 전담 서버를 신규 설계 및 점진 이관

CRMS 규모 확대로 기존 CRMS 서비스 내에 API 기능이 혼재되면서 변경 영향 범위가 커지고 배포/운영 복잡도가 증가했습니다. 이를 해결하기 위해 RESTful 기반 CRMS-API 전담 서버를 분리 구축해 API 영역을 독립 배포 및 확장 가능한 구조로 전환했습니다.

## 목적

- API 책임 분리로 코드베이스 결합도 감소 및 변경 영향 범위 축소
- 독립 배포/확장 가능한 API 서버 구성으로 운영 효율 개선
- 보안(ACL, CORS, 암호화)과 문서화 체계 정비로 운영 품질 강화

## 주요 업무

- RESTful API 서버 아키텍처 분리 및 신규 구축, 운영
- Swagger(OpenAPI) 기반 API 문서화 구축 및 표준 응답 규격 정리
- ACL 강화, CORS 정책 정비, 민감 데이터 암호화 적용
- 공통 Exception Handler 설계 및 에러/응답 처리 표준화
- 기존 CRMS 서비스의 API 기능 이관 및 연동 영향도 관리(점진 이관)

## 문제 해결 과정

- 서비스 로직과 API가 섞인 구조에서 발생하던 사이드이펙트를 API 서버 분리로 차단
- API 변경이 본 시스템에 미치는 영향 범위를 줄이고 배포 리스크를 낮춤
- 보안 이슈를 운영 관점에서 항목화해 개선 → 접근 제어(ACL) 강화, CORS 정책 재정의, 민감 데이터 암호화로 취약점 가능성 최소화
- 문서 부재로 인한 커뮤니케이션 비용을 Swagger로 해소, 스펙 변경 시 추적 가능하도록 문서/규격을 함께 관리

## 성과 / 결과

- API 서버 독립 배포로 변경 영향 범위 축소, 배포 안정성 및 운영 효율 향상
- ACL/CORS/암호화 적용으로 보안 수준 강화 및 운영 리스크 감소
- Swagger 문서화와 표준 응답 체계로 개발/운영 커뮤니케이션 비용 절감
- CRMS 본 시스템 부하 분산 및 장애 확산 가능성 감소
