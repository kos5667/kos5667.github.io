# 상담톡 코드, 아키텍처 현대화 (DDD, TypeScript, 테스트/품질, 보안 통합)

## 개요

- **기간:** 2025.05 ~ 2025.07
- **유형:** 회사 프로젝트
- **인원:** 1명
- **역할:** 기여도 100%

## 프로젝트 소개

> 장기 운영으로 누적된 유지보수성 저하, 의존성 과다, 온보딩 난이도 증가를 해결하기 위해 상담톡 서버 구조를 DDD 기반으로 재설계하고, TypeScript 및 테스트 체계를 정비

장기 운영으로 누적된 유지보수성 저하, 의존성 과다, 온보딩 난이도 증가를 해결하기 위해 상담톡 서버 구조를 DDD 기반으로 재설계하고, TypeScript 및 테스트 체계를 정비했습니다.

## 목적

- 도메인 경계를 명확히 해서 변경 영향 범위를 줄이고 유지보수성을 확보
- TypeScript와 테스트로 런타임 오류를 줄이고 배포 안정성을 높이기
- 업그레이드와 보안 정비로 취약점과 운영 리스크를 줄이기

## 주요 업무

- Layered Architecture → DDD Layered Architecture로 재설계 및 점진 이관
- TypeScript 점진 전환, 핵심 도메인 모듈 타입 안정화
- 전역 exports 중심 구조 제거, 인스턴스 기반 설계로 전환 후 DI(의존성 주입) 기반 서비스로 재구성
- 비대 Service 분해, 인터페이스 기반으로 의존성 역전(DIP) 적용
- Callback 기반 비동기 흐름을 async/await로 전환, 트랜잭션/에러 핸들링 정리
- Jest 기반 테스트 환경 정비, 도메인 단위 통합 테스트 도입(TDD 적용 구간 포함)
- Node 14 → 20 업그레이드 및 EOL 라이브러리 제거/대체, 호환성 이슈 정리
- 소스 내 Secret(계정정보, 암호화 키) 평문 노출 제거, 암호화 적용 방식 정비

## 문제 해결 과정

- 변경 영향이 큰 영역부터 Strangler 방식으로 점진 리팩토링하며 운영 리스크를 통제
- 도메인 단위로 책임을 쪼개고 인터페이스 경계를 세워 결합도/사이드이펙트를 줄임
- 테스트 커버리지를 "핵심 플로우 중심"으로 먼저 확보한 뒤 리팩토링 범위를 확장
- 업그레이드는 사전 검증(호환성, Deprecation) 후 단계 적용으로 다운타임/사고를 예방

## 성과 / 결과

- 타입 체킹과 테스트 기반으로 회귀 리스크를 낮춰 배포 안정성 향상
- 구조 정리(DDD/DI/DIP)로 변경 영향 범위 축소, 유지보수성 개선
- async/await 전환 및 에러 처리 표준화로 장애 분석/대응 속도 개선
- 업그레이드/보안 정비로 취약점 대응 기반 확보 및 운영 안정성 강화
- 코드 일관성/규칙 정착으로 온보딩 비용 절감
